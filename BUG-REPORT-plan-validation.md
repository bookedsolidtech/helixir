# Bug: Plan Validation Fails 100% for Opus — "Plan too short (<100 chars)"

`simpleQuery` in `simple-query-service.ts` (lines 166-171) only captures `text` blocks from assistant responses. Opus 4.6 uses extended thinking by default — it puts the full implementation plan in a `thinking` block and returns only a brief summary in `text`. The PlanProcessor in `lead-engineer-processors.ts` doesn't set `thinkingLevel`, so thinking activates implicitly for Opus. Result: `ctx.planOutput` gets <100 chars, `validatePlan()` rejects it, the feature escalates to BLOCKED, and auto-mode restarts it as "orphaned" — creating an infinite retry loop that burns Opus credits every ~30 seconds.

**Fix options:** (A) Capture `thinking` blocks in `simpleQuery` and use thinking content as fallback when `responseText < 100 chars` — this is the safest fix since thinking content IS the plan. (B) Pass `thinkingLevel: 'none'` in the PlanProcessor's `simpleQuery` call to force Opus to put everything in `text` blocks. (C) Both. Additionally, auto-mode needs a cooldown/guard so it doesn't infinitely restart features that just got BLOCKED by the pipeline, and `pipeline-checkpoint-service` should clear checkpoints when a feature is manually reset to backlog.

**Affected files:** `apps/server/src/providers/simple-query-service.ts` (text extraction ignores thinking), `apps/server/src/services/lead-engineer-processors.ts` (no thinkingLevel set), `apps/server/src/services/auto-mode-service.ts` (infinite retry loop), `apps/server/src/services/pipeline-checkpoint-service.ts` (checkpoints not cleared on manual reset). Confirmed Haiku planning is unaffected — bug is Opus-specific.
